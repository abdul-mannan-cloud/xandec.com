# 02 — Property Lifecycle: The PH-1..PH-Z Phase Model

> Status: DRAFT strawman — [confirm] items pending Jack's review.

This file is the canonical business definition of the phase model Helm already implements. Helm **derives** each property's phase from multi-source signals (Podio, AppFolio, Buildertrend evidence cards tagged `DERIVED`, resolved into a "Hub" `SNAPSHOT` card) — phases are *observed*, not declared by a user. This doc explains what each phase means to the business, who owns the property while it is in that phase, what event moves it in and out, which system should be authoritative at that point, and what typically breaks at each handoff (cross-referenced to the D-rule numbers; the full rule catalog with severities and playbooks is in `06-audit-rules.md`).

Snapshot counts below are from the live tour of 2026-07-06 (1,073 properties total; counts sum exactly).

## 1. The one idea that governs everything

Per Jack, verbatim: *"The opportunity cost of every day that we own a property that isn't rented is 1 day of rent, or 1 day of carry + opportunity cost of our capital. Every day matters and matters the same."*

Consequences for how you should read the phase model:

- **Carry accrues from ownership (the `closed` event / PH-2 entry) onward, in every owned non-earning phase.** PH-2 through PH-7 and PH-10 are non-earning states, and a day spent in any of them costs the same as a day spent in any other. There is no hierarchy of "important" phases.
- **PH-1 is pre-ownership — but not passive.** Contract-to-close is a cycle-time clock but does NOT accrue carry [confirm: carry starts at closing, not contract — right?]. Per Jack (2026-07-09), PH-1 already incurs real out-of-pocket costs on many deals — the contract deposit is tied up (the property's first appearance in QuickBooks), and vacant properties get BG&E in Dominion's name — without accruing carry days under the strawman definition. Renovation *planning* (and, on trustee-vacant deals, actual trash-out/scoping) legitimately starts at contract; detail in the PH-1 section below.
- **PH-8 (Active rental) is the earning steady state** — the state the whole model exists to maximize.
- **PH-9 (Notice / Pre-vacancy) is an earning phase** — the tenant is typically still paying — so its clock is a scheduling/forecast clock, not carry [confirm: do PH-9 days count as carry, given the tenant is usually still paying?].
- The master metric is **days-not-rented (carry days) per property across the whole lifecycle**. Per-phase clocks exist to locate *where* the days accumulate, not to rank which delays matter.
- Clock start/stop definitions per phase, thresholds, and violation mechanics live in `04-cycle-time-model.md`. This file only establishes what the states *are*.

Two loops matter:

1. **Initial pipeline (PH-1 → PH-8):** a property is bought, renovated by DP, handed to DM, leased, and becomes an earning rental. One-way, run once per property.
2. **Turnover loop (PH-9 → PH-10 → back into leasing → PH-8):** run every time a tenant leaves, for the life of the asset. This loop repeats far more often than the pipeline runs, so shaving days here compounds. **The turn re-enters the construction pipeline.** When a managed unit goes vacant, DM turns it over to DP — which performs ALL turnovers — via the **DM→DP turnover email**, and the turn becomes a Podio **Renovation app item with Category = "Turnover"** (per Jack, decision 19 (2026-07-12); see PH-10). A turn is a Renovation item, not a first rehab — same app, distinguished by Category. It then completes via the decision-18 **DP→DM "Handed Off - Rental"** handoff back to leasing. So the full loop is **PH-8 → PH-9 → PH-10 (turn = Renovation Category=Turnover) → PH-4/PH-5 leasing again**. The two turnover emails are opposite ends of this loop and must never be conflated: **DM→DP** starts the turn (vacancy side, decision 19); **DP→DM** ends it (completion side, decision 18). Where a completed turn re-enters is a genuine open modeling question: [confirm: does a completed turn re-enter at PH-4 (ready for leasing) or PH-5 (marketing/vacancy)?]

**How the phase is ACTUALLY derived (per Helm codebase read 2026-07-11, `phase.service.ts:1467-1604`).** The diagram above is the *business* narrative — the flow a property lives. The code does not evaluate that flow; `derivePhase()` is a **first-match-wins priority walk** in this fixed order:

`PH-Z → PH-X → PH-9 → PH-8 → PH-7 → PH-6 → PH-5 → PH-3 → PH-2 → PH-1 → PH-4 → UNKNOWN`

Two consequences that are NOT visible from the state-diagram flow:
- **PH-4 is evaluated LAST, as a near-catch-all:** `is_active AND total_units > 0` (`phase.service.ts:1594-1598`). It is not reached by any "Handed Off - Rental" signal (see PH-3/PH-4 below) — it simply catches any active, unit-bearing property that matched none of the earlier rules. (The engine's own type-header comment claiming a `…PH-5→PH-4→PH-3…` order is stale and does NOT match the code.)
- **PH-9 outranks PH-8** (priority 3 vs 4): any unit on notice would win over occupancy. That is *why* PH-9 would mask PH-8 in principle — but the notice signal is almost never populated today, so in practice PH-9 stays empty (see PH-9).

### PH-1 — Acquisition / Under contract (6)

- **Business meaning:** DP has a signed purchase contract on the property; contract-to-close logistics are running (title, settlement prep). The property is not yet owned — so PH-1 days are a cycle-time clock, not carry (see §1). PH-1 is not passive, though: renovation *planning* starts at contract (see the record-creation and pre-closing-work bullets below), and the deal incurs real out-of-pocket costs — deposit tied up, BG&E for vacants — without accruing carry days under the strawman carry definition.
- **Owner:** DP Acquisitions — Brian Leibowitz (Acquisitions Director) sources the deal; Victoria Robinson (Transaction Coordinator) runs contract-to-close.
- **Entry driver:** purchase contract signed (business event). Oddly, the driver string observed on the PH-1 evidence card is `Closed` ("Acquisition / Under contract · 4/21/2025 · Closed") — closing is the PH-1 *exit* event, so the derivation may assign drivers/dates to early phases retroactively [confirm: how does the derivation assign drivers/dates to early phases?]. Note the literal `contract_signed` driver string belongs to PH-7 (lease execution), not PH-1.
- **Record creation at contract (per Jack, 2026-07-09):** the property's data spine starts as three near-simultaneous births —
  1. a **Buy Closings** item in Podio — the under-contract record Victoria runs (see "System of record" below for the naming reconciliation);
  2. a Podio **Renovation app** item — auto-created by a Podio **GlobiFlow** automation when the Buy Closings item is first created (per Jack, 2026-07-09; the GlobiFlow `item.create` webhook on Buy Closings is confirmed active, per Podio API tour 2026-07-09) — which puts the property on the renovation-prep radar. Default status when the property is vacant: **"Design & Scope"**; **"Trash Out"** may also be selected when the property contains abandoned personal property, so possessions can be cleared and the property scoped. Both are values of **Renovation.Phase** — a 43-stage vocabulary spanning the entire property life (Asset Preservation → … → Trash Out → Design & Scope → … → Handed Off - Rental → … → Sold/Rented/Write Off/Dead), far finer-grained than this doc's phase model; full list in `09-podio-architecture.md`;
  3. the contract **deposit entry in QuickBooks** — a deposit is required on almost all purchase contracts, and its accounting entry is the FIRST place a property shows up in QuickBooks, during PH-1, pre-closing.
  The Buildertrend job, AppFolio unit, and BNA asset are all born later (see `05-data-sync-map.md`).
- **Pre-closing work — the seller-type fork (per Jack, 2026-07-09):** vacant properties get **BG&E** (utilities) in Dominion's name so the property can be scoped correctly. How much can happen before closing depends on the seller:
  - **Trustee purchase (courthouse steps):** if the property is found vacant, it is treated as abandoned — Dominion changes the locks, puts BG&E in its own name, and trashes it out BEFORE closing. Scope/design work runs during PH-1.
  - **Private-individual seller:** utilities are NOT put in Dominion's name and there is NO trash-out until AFTER closing, to avoid the drama of accidentally removing items with sentimental value to the seller.
  Seller type is therefore a real modeling attribute — it determines what work is allowed pre-closing, so downstream clock expectations differ by acquisition type (see PH-2 below and `04-cycle-time-model.md`).
- **Exit driver observed:** `closed` (settlement) → PH-2.
- **System of record:** Podio — Victoria populates the **Buy Closings** app (app 9068504, 1,139 items), which lives in the **Closings** workspace alongside Sale Closings, per Podio API tour 2026-07-09. (Earlier docs inferred a workspace called "Acquisitions & Titles" — no such workspace exists; Helm's tour label "Closing" under the "Acquisitions CRM" banner maps to this Closings workspace. Full architecture in `09-podio-architecture.md`; authority rules in `03-systems-of-record.md` §4.) Note the acquisition-type fork below already exists as a field: **Buy Closings.Type** = Private | Trustee | REO | Builders | Short Sale | OHAAT | NCST; the **EMD** field is the deposit, and **Purchasing Entity** the buying entity. This is the one lifecycle stage where Podio is genuinely primary, not a mirror. Note Jack's direction that Podio is a legacy source Helm should eventually absorb (see `05-data-sync-map.md`); Helm-native acquisition records are the long-term replacement.
- **What goes wrong at the handoff:** the closing itself is well-controlled (settlement statements feed cost basis via the Controller); the failure mode is downstream — see PH-2/D1.
- **Edge case:** [confirm: do some acquisitions arrive already-tenanted (or rent-ready) and skip PH-3 renovation — or even land directly in PH-8?] If yes, the derivation logic needs a legal PH-2→PH-4/PH-8 shortcut rather than treating it as missing evidence.
