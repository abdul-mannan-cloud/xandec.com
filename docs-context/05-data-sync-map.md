## 1. The problem Helm exists to kill: the double-keyed status update

Jack has named the duplicate-entry burden precisely, and it is the single highest-value target for Helm's data-movement work:

- **During construction**, staff must update project statuses in **both Podio and Buildertrend**.
- **During turnover & leasing**, staff must update statuses in **both Podio and AppFolio**.

In both cases the *operational* system (Buildertrend for construction, AppFolio for property management) is where the real work is managed. The **duplicated keystroke is the status update into Podio** alongside the operational system. Podio is the "gap-filler" middleware (the company guide's own term) — it holds workflow and compliance records the operational systems don't capture, and that role forces staff to mirror status into it constantly.

**Which side is the source is now fixed (per Jack, decision 18, 2026-07-12).** Jack has named the direction of both mirrors explicitly: during renovation (PH-3) **Buildertrend is the source and the Podio Renovation record MIRRORS it** — the sync flows BT→Podio, and a Podio status that disagrees with Buildertrend progress is a stale mirror, not an independent fact. After the renovation→leasing handoff (PH-4+) **AppFolio is the primary source for day-to-day management and Podio drops to a compliance supplement** for what AppFolio can't track. So the "duplicated keystroke into Podio" is, in both segments, a keystroke into the *non-authoritative* mirror — which is exactly why Helm can eventually stop asking humans to make it (§5). (This is Jack's INTENT; the code gap that keeps Helm from seeing the PH-3→PH-4 handoff today is preserved and documented in §3.6.)

Every *discrepancy* finding in Helm today (363 of the 423 open findings; see `06-audit-rules.md`) is, at root, evidence of this same pattern: a fact was born in one system and a human either never copied it to the sibling system, or copied it late, or copied it wrong. (The other 60 findings — the D17/D18 compliance expiries — are a different failure class: real-world lapses of licenses and lead-paint certs, not propagation failures.) The discrepancy catalog is a map of where manual propagation breaks; this file is the map of what *should* propagate.

> **Reconciliation caveat (per Helm codebase read 2026-07-11):** the finding counts throughout this file (423, 363, 200, 135, …) are **point-in-time query results, not a stored backlog.** Helm recomputes all rules' SQL live on every read and persists **no** findings — `detected_at` is the request timestamp, and there is no dismiss / snooze / resolve / aging state, so the doc's "open findings" phrasing is loose (there is no open/closed status). Separately, the code ships **9 discrepancy/compliance rules, not the 7 the tour surfaced**: D5 (Podio write-off but AppFolio still active, MED) and D9 (unit vacant >14d with no turnover, HIGH) also run but did not appear in the tour's counts. Both facts are threaded into the sections below (`audit.service.ts:39-102`; see `06-audit-rules.md`).

One sharpening from the complete GlobiFlow flow inventory (per GlobiFlow UI tour 2026-07-09; flow names verbatim in `sources/podio-architecture.md` §6): **inside Podio, propagation is heavily flow-automated** — cross-app record creation and field sync are the norm, not the exception ("Update Renovation App Closing Date" from Buy Closings; "Update Listings on Approved Applications"; "Update Construction Rental Status" ×3 writing leasing status back into the construction record; HAP's four-app fan-out "Update Pre-Contract, Inventory, Inspection, and Listings on Rented Status"; "Lead Paint Compliance SYNC"; Bank Loans add/update from both Buy Closings and Lead Certs). The manual propagation Helm kills is therefore specifically the **cross-platform** edges — Podio ↔ AppFolio, Podio ↔ Buildertrend, Podio/Buildertrend → QuickBooks/BNA — where no automation reaches. That is largely where the discrepancy catalog points: most D-rules compare across platforms. (D11 is the Podio-internal exception — Vacant Units.Owner vs Lead Certs & Water Bills.Owner is an edge GlobiFlow could reach but simply has no flow covering; its 200 findings are the cost of that gap.)

## 2. The property-record spine: one asset, six births — three of them at contract

A property's record is "born" in six systems as it moves through the lifecycle. Per Jack (2026-07-09), the first three births are **near-simultaneous, at contract** (PH-1, pre-closing); the remaining three follow one at a time, each a manual keying event seeded from what came before:

```
PH-1 At contract — three near-simultaneous births
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐
│ Podio            │   │ Podio            │   │ QuickBooks       │
│ Buy Closings     │   │ Renovation app   │   │ deposit entry    │
│ item             │   │ item             │   │ (the property's  │
│ (contract-to-    │   │ ("Design & Scope"│   │ FIRST ledger     │
│ close record)    │   │  / "Trash Out")  │   │ appearance)      │
└──────────────────┘   └──────────────────┘   └──────────────────┘
         │
         v
PH-2 Closing handoff         PH-4/5 Rent-ready/Marketing      In-service
┌──────────────────┐        ┌──────────────────┐        ┌──────────────────┐
│ Buildertrend     │  --->  │ AppFolio         │  --->  │ BNA              │
│ Job              │        │ Property / Unit  │        │ Fixed Asset      │
│ (job costing,    │        │ (owner acct,     │        │ (cost basis,     │
│ scopes, POs,     │        │ listing, tenant  │        │ depreciation —   │
│ bills)           │        │ ledger)          │        │ accounting team) │
└──────────────────┘        └──────────────────┘        └──────────────────┘
```

- **Podio Buy Closings item** — the property record originates here when the property goes under contract. Buy Closings is an **app in the Closings workspace** (app 9068504, 1,139 items; no "Acquisitions & Titles" workspace exists — per Podio API tour 2026-07-09, see `03-systems-of-record.md` §4). Victoria Robinson (Transaction Coordinator, DP) populates it through contract-to-close, including settlement statements (ALTA/HUD-1s), deeds, and closing cost breakdowns. The app already carries the spine's key attributes as first-class fields (per Podio API tour 2026-07-09): **EMD** (the earnest-money deposit — the very amount that is QuickBooks' first touch), **Contract Date**, **Closing Date**, **Type** (Private | Trustee | REO | Builders | Short Sale | OHAAT | NCST — the trustee-vs-private acquisition fork already encoded as data), and **Purchasing Entity** — so Helm can read the deposit, the spine's anchor dates, the acquisition type, and the owning entity directly from Buy Closings, with no new data capture.
- **Podio Renovation app item** — auto-created by a Podio **GlobiFlow** automation when the Buy Closings item is first created (per Jack, 2026-07-09); puts the property on the renovation-prep radar. Default status when vacant: **"Design & Scope"**; **"Trash Out"** may also be selected when the property holds abandoned personal property. For trustee (courthouse-steps) purchases found vacant, scoping and trash-out legitimately happen pre-closing; for private-individual sellers that work waits until after closing (see `02-property-lifecycle.md`).
- **QuickBooks deposit entry** — almost every purchase contract requires a deposit, and accounting for that deposit is the FIRST place the property appears in QuickBooks — during PH-1, before closing. QuickBooks is therefore not a purely post-closing consumer: its first touch predates every birth except the two Podio items it accompanies.
- **Buildertrend job** — born at the closing → construction handoff (PH-2). Created manually from the Podio/closing data [confirm: is the Buildertrend job keyed by the PM, by Victoria, or by James Stewart's team — and is any of it templated, or fully re-keyed?].
- **AppFolio property/unit** — born when the property approaches rent-ready/marketing (PH-4/PH-5) and DM takes over. Set up as a unit under the correct owner account. Entry is manual re-key from upstream records [confirm: who creates the AppFolio property/unit record — Stephanie's leasing team or a DM admin?].
- **BNA fixed asset** — born at in-service. The Controller establishes cost basis from the settlement statements Victoria filed in Podio, plus capitalized renovation costs from Buildertrend. Manual keying by the internal accounting team (stated policy: accounting maintains BNA directly, not the CPA).

**The new pre-closing propagation edge.** One real-world event — "went under contract" — lands in three places today: → Buy Closings (manual), → Renovation app (**auto-created by a Podio GlobiFlow automation** when the Buy Closings item is first created — per Jack, 2026-07-09; the underlying `item.create` webhook on Buy Closings is confirmed live via the Podio API, registered 2020 — per Podio API tour 2026-07-09; the flow itself is now identified by name — **"Add new property to construction pipeline"** — with sibling flows on the same app auto-creating records in Lead Paint Compliance (year-built-gated) and Bank Loans Pipeline — per GlobiFlow UI tour 2026-07-09), → QuickBooks deposit entry (accounting). The Podio→Podio hop is therefore NOT a duplicate-entry seam — it's already automated. Three consequences: (1) a "Buy Closings item with no Renovation counterpart" gap rule is still worth having, but it monitors *automation failure*, not human forgetfulness; (2) GlobiFlow automations like this one are load-bearing process steps that Helm must replicate when it absorbs the apps they connect (see §5 risks); (3) the contract birth is not the only flow-created Renovation item — turnover projects in Pipeline - Construction are likewise auto-created from the turnover apps — the live path is the newer Occupancy generation (Sched Vacants), whose DP/3P flows fire on `Turn Over Status = "Turn Over to DP"`; the older generation's Upcoming Turnovers flows are **DISABLED** (struck-through in the GlobiFlow UI 2026-07-12, per Jack decision 19), and its apps are an archive retired ~November 2017 (per Podio API activity probe 2026-07-11) — so the older path is not merely dormant, it is switched off; see §3.5).

**The turnover propagation edge (PH-3→PH-4).** The AppFolio birth above is not a lone keying event — per Jack, decision 18 (2026-07-12) it is one leg of a **three-part synchronized state change** fired by the **DP→DM property-turnover email** (Dominion Properties handing a finished property to Dominion Management for leasing): (1) the Buildertrend job is marked **complete**, (2) the Podio Renovation phase is set to **"Handed Off - Rental"**, and (3) the property is **created in AppFolio as an asset under the appropriate owner**. This is the turnover analogue of the contract triad at the top of this section, and the **AppFolio-asset-creation is the concrete PH-4 event.** Because the three legs must move together, **any leg present without the others is a discrepancy**: Podio "Handed Off - Rental" but no AppFolio asset = handed off but never onboarded; AppFolio asset but Podio still in an in-progress reno phase = mirror lag; Podio "Handed Off - Rental" but the BT job not complete = premature flip — state-consistency rules now proposed as strawman candidates in `10-common-data-map.md` §4. **Code-reality caveat (preserved):** Helm ingests all three record legs, but `derivePhase()` ignores "Handed Off - Rental" (§3.6), so Helm cannot currently detect this handoff from the Podio leg — a real gap versus Jack's intended definition of PH-3→PH-4, not just unused granularity. Note this DP→DM email is the *renovation-complete* handoff and runs opposite to the **DM→DP** move-out turnover email in the §3.5 flow chain (which turns a unit *back* into construction after a tenant leaves); the two must not be conflated. [confirm: is the DP→DM renovation-complete turnover email a discrete signal Helm can ingest — or a GlobiFlow-generated message — or must Helm infer the handoff purely from the three record legs?]

**The turn-loop START edge (PH-8→PH-9→PH-10) — the OTHER turnover, opposite in direction.** The completion triad above is only one end of the turn loop; per Jack, decision 19 (2026-07-12) its *other* end is the **DM→DP turnover** — the vacancy-side turn trigger. When a *managed* unit goes vacant, Dominion Management turns it over to Dominion Properties (DP performs ALL turnovers) to make it rent-ready again — the mirror image of the DP→DM renovation-complete handoff above. The operational trigger (GlobiFlow UI 2026-07-12) is **`Sched Vacants.Turn Over Status` changing to "Turn Over to DP"** (Sched Vacants app, Occupancy workspace), which fires its own three-part fan-out: (1) a **Buildertrend job** is created for the turn (per Jack); (2) a Podio GlobiFlow **creates a new Renovation-app item with Category = "Turnover"** (DP flow ID 614403, gated on `Owner = "Dominion Properties"`; the third-party owner-split variant sets Category = "3rd Party - Turnover"); (3) a sibling **"turnover email from DM to DP"** flow sends the scoping email (GlobiFlow-generated — `no-reply@automation.podio.com`, "From: Tori Fowlkes" (DM Field Inspector — not Victoria) — broadcast to a ~19-person DP+DM distribution) carrying rich last-tenancy detail, including full HVAC/plumbing/electrical/roofing history with vendor + cost per line, the recurring-issue/capex signal (example captured in `sources/example-turnover-email-dm-to-dp.md`). Because the Renovation app hosts BOTH initial rehabs AND turnovers, distinguished by Category, **a turn is a Renovation item with Category = Turnover** — the turn loop re-entering the construction pipeline, not a first rehab; it then completes back through the decision-18 DP→DM handoff ("Handed Off - Rental" + AppFolio asset) into leasing. So the two edges **bracket the turn loop and must never be conflated: DM→DP (decision 19) is the START, DP→DM (decision 18) is the END** — two turnover emails, opposite directions, at the two ends of the loop. **Code-reality caveat (preserved):** Helm does NOT read the Sched Vacants app (per Helm codebase read 2026-07-11 — zero references to Sched Vacants or app 19831048), so the operational trigger for the ENTIRE turn loop — `Turn Over Status = "Turn Over to DP"` — is invisible to Helm today. The operational reality exists and is rich; Helm simply cannot see it, which is why PH-9/PH-10 don't populate and Helm cannot yet detect turnovers at all. Wiring Sched Vacants in is the net-new work that would make the turn loop visible.

**The disposition propagation edge (PH-X) — the sale end of the property life.** Just as the contract triad opens the property's life (§2 top) and the two turnover edges bracket the turn loop, per Jack, decision 20 (2026-07-12) a fourth synchronized edge closes it: the **disposition (sale) pair**. When a property is going to be sold, its **Renovation.Phase** walks a three-state machine — **On Market** (listed for sale) → **Under Contract** (offer accepted) → **Sold** (closed) — and a **Sale Closings item is created when the property goes Under Contract** (one per under-contract property). At close **both** flags flip together: Renovation.Phase = "Sold" AND Sale Closings.Status = "Sold". Renovation.Phase and Sale Closings.Status are therefore a **synchronized pair** — the disposition analogue of the turnover triad — with expected invariants: Renovation "Under Contract" ⇔ a Sale Closings item exists (Status Under Contract), and the two "Sold" flags move as one. The out-of-sync combinations (Renovation "Under Contract" with no Sale Closings item; Sale Closings "Under Contract" without the matching Renovation phase; the two "Sold" flags disagreeing) are proposed as strawman state-consistency candidates in `10-common-data-map.md` §4, with the existing candidate **D21** ("Sale Closings Sold but AppFolio still `is_active`") now confirmed as the disposition *end-state* check. This is the **PH-X (disposition) arm — distinct from PH-Z (write-off)**: On Market and Under Contract are disposition-IN-PROGRESS states and Sold is terminal, which is why the Renovation app spans the ENTIRE property life (acquisition prep → construction → turnover → disposition), not just construction — the breadth its 43-value Phase vocab already carries (§3.6). **Code-reality caveat (preserved, per Helm codebase read 2026-07-11):** live `derivePhase()` does NOT read Renovation.Phase "On Market" / "Under Contract" / "Sold" — none are in the write-off list and none match `/complete/i`, so all three fall into the **PH-3 catch-all** (the same shallow read documented in §3.6). Helm's PH-X fires only on compliance **`lifecycle_stage='disposed'` AND `is_active=false`** (sourced from the Podio compliance boards' property-status, not from Renovation or Sale Closings). So a property that is On Market / Under Contract / Sold in Renovation is largely invisible to Helm's live disposition detection unless the separate compliance property-status is also set to disposed and AppFolio is deactivated — reading Renovation.Phase and/or Sale Closings.Status is net-new work to make dispositions visible. [confirm: is the Sale Closings app in Helm's Podio manifest allowlist (ingested to a table), or unmanifested and silently skipped — i.e. does Helm even hold the Sale Closings data that live disposition detection would read?]

Downstream of the spine: the entity ledgers (QuickBooks Desktop per entity today, Sage single-database after the 2026 migration) consume this data but do not write back — with the wrinkle above that QuickBooks' first entry happens at contract (the deposit), not at closing. BNA and the ledgers are **downstream consumers only** — Helm never needs to sync *from* them, but the accuracy of what flows *into* them depends on the upstream chain being right.

Helm's own place in this picture is purely observational: it **ingests three of the six systems read-only** — Podio via an org-wide OAuth crawl, AppFolio via the Reports API, Buildertrend via headless Playwright scrapers — and **writes to none of them** (per Helm codebase read 2026-07-11; see `03-systems-of-record.md` §7). It watches these births happen; it does not create them and cannot yet correct them (this is the hard prerequisite for §5's write-back roadmap).

The audit rule guarding the spine's weakest link today: **D1 (HIGH) — "Closing complete but no Buildertrend job"** (3 open findings). The equivalent gap rules for the AppFolio and BNA births do not exist yet — see candidate rules in `06-audit-rules.md`.

## 5. PROPOSAL — the Podio-retirement roadmap

> Everything in this section is a proposal for discussion, not a spec. The direction, however, is Jack's stated intent: **Helm becomes the middleware and retires Podio completely, eventually.** Design consequence (also Jack's): treat Podio as a legacy source being absorbed — new Helm features should not deepen Podio dependency, and Helm-native records should be able to take over the roles Podio plays. The retirement unit is the **app (or app cluster)**, not the workspace: the real inventory (per Podio API tour 2026-07-09) is 15 workspaces / ~65 apps, and the roles earlier drafts called "vacant units," "rental billing," and "compliance & licensing" map to the Occupancy apps and to one combined app (Lead Certs & Water Bills) — see the re-mapped stages below.

**Hard prerequisite for Stages 2–3 (per Helm codebase read 2026-07-11): the write-back this roadmap assumes DOES NOT EXIST YET.** Helm is entirely **read-only inbound** today — it ingests Podio, AppFolio, and Buildertrend and writes to none of them; every remediation the D-rules produce is an instruction for a human to key by hand (`audit.service.ts` / `phase.controller.ts` expose reads and one internal snapshot only). Before any Stage-2 "fix it" button or Stage-3 "write once in Helm" behavior can work, a **write path to Podio (and ideally AppFolio) must be built from scratch** — that plumbing is the true first task of Stage 2, not an incidental requirement. Stage 1 below is the only stage actually built. The roadmap remains the proposal it always was.

### Stage 1 — Detect (BUILT)

The D-rule audit catalog is this stage: **9 discrepancy/compliance rules** (the tour surfaced 7 firing and 423 findings; D5 and D9 also run in code — per Helm codebase read 2026-07-11), per-property source-presence dots, and the phase-derivation engine that reconciles conflicting signals. Helm already knows *where* propagation failed — though only **at read time**: findings are recomputed live on every request and never persisted (no stored backlog, no history), so "Stage 1" is a live query surface, not a tracked queue. Nothing to build here except more rules (see candidates in `06-audit-rules.md`).

### Stage 2 — One-click remediation from a finding

Turn each finding from a report row into an action button. The finding already contains everything needed to fix it:

- **D1** → "Create Buildertrend job" pre-filled from the Podio closing record. If Buildertrend can't be written programmatically (§4), the button instead produces a filled-out worksheet/checklist so the human keying it re-keys nothing — they transcribe from one screen, once.
- **D7** → "Close out Podio Vacant Unit," pre-filled with the AppFolio move-in date.
- **D14** → "Create Podio Vacant Unit entry" from the approved AppFolio application.
- **D11 / D20** → "Adopt value from [authoritative system]" pushed to the disagreeing copy, once the authority questions in §3.2/§3.3 are confirmed.

Stage 2 requires write access to Podio (the public API exists, but **Helm has built no write path to it — or to any platform — today; it is read-only inbound, per Helm codebase read 2026-07-11**) and ideally AppFolio. Building that write path is Stage 2's actual first deliverable. It does not eliminate double entry — it eliminates the *cleanup* burden and proves out write-back safely, on records a human explicitly reviewed.

### Stage 3 — Helm becomes the status hub, then Podio dies workspace by workspace

The end state for §1's double-keyed status: **staff update status ONCE, in Helm.** Helm propagates to the operational system where needed (AppFolio via API; Buildertrend via API-or-worklist per §4) and to Podio for as long as Podio remains alive. During that window Helm *writes* Podio instead of humans doing it — the duplicated keystroke dies immediately, even before Podio does. (All of this presumes the Stage-2 write path exists; none of it is possible while Helm remains read-only inbound — per Helm codebase read 2026-07-11.)

Then decommission Podio one app (or app cluster) at a time, each with a parallel-run period (Helm-native record is authoritative; Podio copy is machine-maintained; consumers — especially accounting — verify they can live off Helm before the app is frozen). Strawman order, re-mapped to the real app inventory (per Podio API tour 2026-07-09), chosen to bank the biggest finding volumes first and to leave the accounting-critical app for last [confirm: is this the right order?]:

1. **Turnover & leasing apps — the LIVE leasing surface: Sched Vacants + Vacant Units (Occupancy workspace)**, with 36 and 94 new items respectively in the last 90 days and newest items created this week (per Podio API activity probe 2026-07-11). Highest audit pain (D7 + D14 + half of D11); contained blast radius. The which-generation question earlier drafts carried here is now RESOLVED empirically: the **older Move Ins & Move Outs generation** (Listings, Rental Applications, Upcoming Turnovers, Move Outs, HAP Appointments, Leasebacks, Pre-Contract Inspections) is an **ARCHIVE, retired ~November 2017** — no item created in any of its apps since 2017 (per Podio API activity probe 2026-07-11). The two generations' overlapping shapes (Rental Applications vs Applications; Upcoming Turnovers vs Sched Vacants) are therefore data-archaeology, not a live duplicate-entry risk inside Podio — there is no active cross-generation keying to retire. The older workspace accordingly **drops out of this stage as a flow-replication target and becomes a data-migration/archive decision instead** (what history to preserve, and where). Its structural flows — Upcoming Turnovers' vacancy click auto-creating Move Outs records and Pipeline - Construction projects — are **DISABLED** (struck-through in the GlobiFlow UI 2026-07-12, per Jack decision 19), on top of the apps being an archive with no new items since ~November 2017 (per Podio API activity probe 2026-07-11); so the live turn engine is solely the newer Occupancy generation, and those older flows are already switched off — nothing to replicate. 2. **Lead Certs & Water Bills (rent copy + compliance, one app) and its satellites** — the "Rental Billing" and "Compliance & Licensing" stages of earlier drafts collapse into this single app (1,308 items), because rent comparisons (D20) and license/lead-cert expirations (D17/D18) live in the same records (per Podio API tour 2026-07-09); the app cannot be half-retired, so both roles must land in Helm together. - **Write-back asymmetry (the big one) — on top of the fact that write-back is currently *zero*.** Baseline first: Helm writes to no platform today (read-only inbound, per Helm codebase read 2026-07-11), so the whole roadmap is gated on building write paths at all. ## 6. What this means for Helm's data model (informative, not prescriptive)

- Helm already derives phase from multi-source evidence; the sync map says Helm should also track, per data element per property, **which copy is authoritative and which copies are derived** — that's the schema Stage 2 buttons and Stage 3 propagation both hang off.
- Every element in §3 has an "authority on conflict" answer; three of them are unconfirmed ([confirm] flags in §3.2, §3.3, §3.7). Those three answers are prerequisites for any auto-remediation.
- **Segment-level authority is now DECIDED even though per-field conflict-resolution stays deferred.** Decision 16 deferred all "wins on conflict" calls to a later phase; decision 18 (per Jack, 2026-07-12) carves out the one authoritative slice for the construction→leasing spine by naming the **primary-source custodian per lifecycle segment** — PH-3 renovation: Buildertrend is source / Podio Renovation mirrors; PH-4+ post-handoff: AppFolio primary / Podio compliance-supplement. So the "which copy is authoritative" schema above already has its coarse (segment) axis filled in for this spine; what remains deferred per decision 16 is the fine (per-field) tie-breaker. The two decisions reconcile rather than conflict: **18 sets the custodian, 16 still owns the field-level rule.**
- The cycle-time model (`04-cycle-time-model.md`) depends on this file indirectly: phase transitions are only as trustworthy as the propagation that evidences them. Killing duplicate entry is also a data-quality investment in the carry-day clock.
