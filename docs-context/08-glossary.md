## 2. Property lifecycle & operations terms

**Buy Closings (Podio app)** — The Podio app a property is entered into when it goes under contract — the first of the three near-simultaneous births of a property's data spine at contract (Buy Closings item + Renovation app item + QuickBooks deposit entry; see 05-data-sync-map.md). It is an app (1,139 items) in the **Closings** workspace, alongside Sale Closings — per Podio API tour 2026-07-09; earlier drafts of this doc set inferred an "Acquisitions & Titles" workspace, which does not exist. Key fields already present: **Type** (Private | Trustee | REO | Builders | Short Sale | OHAAT | NCST — the acquisition/seller-type attribute), **EMD** (the deposit — see §3), **Purchasing Entity**, Contract Date, Closing Date, Date Ratified. A GlobiFlow `item.create` webhook is confirmed active on this app (the Renovation auto-creation trigger). See 02-property-lifecycle.md PH-1 and 09-podio-architecture.md.

**Renovation app (Podio)** — The Podio app whose item is auto-created (by a GlobiFlow — see below; `item.create` webhook confirmed active) when a Buy Closings item is first created, putting the property on the renovation-prep radar before closing. Sole app (2,346 items) in the **Pipeline - Construction** workspace. Its **Phase** field is a 43-stage pipeline spanning the entire property life — Asset Preservation → … → Trash Out → Design & Scope → … → Demo → … → Punchout → Clean → Staging & Photos → 30 Day Closeout & Bill → **Handed Off - Rental** → On Market → … → Sold/Rented/Write Off/Dead, plus new-build stages — far finer-grained than Helm's single PH-3 (per Podio API tour 2026-07-09). "Handed Off - Rental" is a Renovation.Phase value that originates in Podio, not Buildertrend — but per Helm codebase read 2026-07-11 it does NOT drive Helm's live phase: `derivePhase()` never matches the string, so a Handed-Off property falls into the PH-3 catch-all in current-phase logic. "Handed Off - Rental" produces a PH-4 card only in the retrospective journey deriver (`deriveHistoryFromDates`) and appears in D9's turnover-exclusion list. This supersedes the earlier Podio-API-tour reading that treated "Handed Off - Rental" as the live PH-3→PH-4 driver — see 02-property-lifecycle.md. The Renovation item is also the parent of the 18 per-trade job apps in the Construction Scheduling workspace (see 09-podio-architecture.md). Default Phase for a vacant property at contract is "Design & Scope"; "Trash Out" may also be selected when the property contains abandoned personal property. **Confirmed (Jack, 2026-07-09): this app is the Podio side of the construction double-entry** — the place staff double-key project statuses alongside Buildertrend. Per Jack, decision 18 (2026-07-12), within that double-entry **Buildertrend is the source and the Renovation data mirrors it** — so a Renovation.Phase that disagrees with Buildertrend progress is the stale mirror (BT wins), not an independent truth. The app hosts BOTH initial rehabs AND turnovers, distinguished by its **Category** field — a Category = "Turnover" (or "3rd Party - Turnover") item is a vacancy-side turn, not first-time construction (per Jack, decision 19, 2026-07-12; GlobiFlow UI 2026-07-12): the turn loop re-entering the construction pipeline. So "it's in the Renovation app" does not by itself mean initial construction — see "Turnover / turn" and "Turn Over Status." See 02-property-lifecycle.md and 05-data-sync-map.md.

**GlobiFlow** — Podio's workflow-automation engine (marketed as "Podio Workflow Automation"). Dominion's Podio is not just passive data: **~276 flows** were inventoried via the GlobiFlow UI on 2026-07-09 (names verbatim in sources/podio-architecture.md §6; the Occupancy workspace alone carries 128). Four classes: (1) **record-creation chains** — load-bearing steps like Buy Closings → Renovation auto-creation and the flow-driven turnover chain (vacancy click → Vacant Units record → Renovation turnover project → lead-cert renewal task → DM→DP handoff email; the parallel older-generation Move Outs leg is vestigial — see below); (2) **~120 per-person hardcoded notifications** ("notify Kenesha when RFTA is submitted") — Podio's brittle push-alert layer, which Helm's role-based alerting replaces wholesale; (3) **deadline watchdogs** (RFTA-late, LLF-expiration, inspection-15-days-out) — proto-T/D-rules; (4) **cross-app field syncs** (e.g., leasing statuses written back into the Renovation record) — Podio-internal edges are heavily automated; the truly manual edges are cross-platform (Podio ↔ AppFolio/Buildertrend/QuickBooks). Both leasing-app generations (Upcoming Turnovers/Move Outs AND Sched Vacants/Vacant Units) are wired with structural flows — per GlobiFlow UI tour 2026-07-09 — but only the newer generation fires in practice: the Move Ins & Move Outs workspace is an archive with no new items since ~Nov 2017, so its flows are vestigial (flows fire on item events, and no new items arrive) and should be retired, not replicated — per Podio API activity probe 2026-07-11. The ~276 headline therefore overstates the live estate: Occupancy's 128 includes 54 flows on the Applications app (dormant since 2022-05, per the same probe); the active leasing-flow core is roughly Vacant Units (67) + Sched Vacants (7). Flow-level trigger/action logic is still uncaptured; any Podio-retirement plan must replicate a live app's flows before freezing it — dead-app flows get retired, not replicated (see 05-data-sync-map.md §5 and 09-podio-architecture.md).

**Design & Scope** — The default Renovation-app status for a vacant property at contract: the property is being scoped and its renovation designed during PH-1, before closing. Real pre-closing work with real out-of-pocket costs (deposit tied up, BG&E service) that nevertheless accrues no carry days under the strawman carry definition, since carry starts at closing. See 02-property-lifecycle.md.

**Trash Out** — A Renovation-app status selected when a property contains abandoned personal property that must be cleared before it can be scoped. Timing follows the seller-type fork: on trustee purchases found vacant the property is treated as abandoned and trashed out BEFORE closing; when buying from a private individual, trash-out waits until AFTER closing to avoid removing items with sentimental value to the seller. See "Trustee purchase" below and 02-property-lifecycle.md.

**Trustee purchase / courthouse-steps acquisition** — An acquisition bought from a trustee (foreclosure sale, "courthouse steps"). House doctrine: if a trustee-purchased property is found vacant it is considered abandoned — Dominion changes the locks, puts BG&E in its own name, and trashes it out before closing. Private-individual sellers get the opposite treatment (no utilities in Dominion's name, no trash-out until after closing). Modeling consequence: seller type changes what work is allowed pre-closing, so clock expectations differ by acquisition type — trustee-vacant deals should hit job start faster after closing because scope/trash-out is already done. The attribute already exists as a field: **Buy Closings.Type** = Private | Trustee | REO | Builders | Short Sale | OHAAT | NCST (per Podio API tour 2026-07-09). See 02-property-lifecycle.md and 04-cycle-time-model.md.

**BG&E (Baltimore Gas & Electric)** — The local utility. Vacant properties under contract get BG&E service activated in Dominion's name so the property can be scoped correctly — a scoping prerequisite and one of PH-1's real pre-closing costs. Timing follows the seller-type fork: before closing on trustee-vacant purchases, after closing when the seller is a private individual. See 02-property-lifecycle.md.

**Turnover / turn** — The work cycle triggered when a managed unit goes vacant: repair, clean, and refresh it back to rent-ready condition. **DP performs all turnovers** — a newly-vacant managed unit is turned over BY DM TO DP to be made rent-ready (per Jack, decision 19, 2026-07-12), the vacancy-side START of the turn loop and the opposite direction from the decision-18 DP→DM completion handoff. Operationally the turn fires from **Sched Vacants.Turn Over Status → "Turn Over to DP"** (per GlobiFlow UI 2026-07-12; see "Turn Over Status"), which spawns a Renovation item with **Category = "Turnover"** plus a Buildertrend job and the DM→DP turnover email — so a turn re-enters the construction pipeline as a Renovation item (the Renovation app hosts both initial rehabs and turnovers, distinguished by Category). Each turnover is tracked as an independent job in Buildertrend, and turnover labor/material costs are typically routed as a DM operating expense (repairs & maintenance) rather than capitalized — see "Capitalized vs expensed work." Helm cannot currently see the turn: it does not read Sched Vacants, so the trigger is invisible and PH-9/PH-10 stay empty in Helm (per Helm codebase read 2026-07-11) — see 02-property-lifecycle.md.

**Sched Vacants (Podio app)** — The notice-phase (PH-9) record (1,254 items, Occupancy workspace): tenant, Move Out Date, Move Scenario and Reason Leaving vocabularies, per-trade pre-turn review fields (PLUM/HVAC/ELEC/ROOF). Its **Turn Over Status** field (Turn Over to DP | Not Turning Over - Parked) is both the turn trigger and the turn-hold decision: "Turn Over to DP" fires the vacancy-side turn loop (see "Turn Over Status" and "Turnover / turn"), while "Not Turning Over - Parked" holds the unit (see the parked entry below). Together with Vacant Units it is the live leasing/turn surface — 36 new items in the last 90 days, newest 2026-07-10 — and the pair is also the live PH-10 record (per Podio API activity probe 2026-07-11). The older-generation Upcoming Turnovers app in Move Ins & Move Outs covered the same ground but is an archive (no new items since Nov 2017, per the same probe), so the earlier duplicate-entry risk is resolved — there is no active cross-generation keying. Per Podio API tour 2026-07-09; see 09-podio-architecture.md.

**Turn Over Status (Sched Vacants field) — the turn trigger** — The Sched Vacants field whose value **"Turn Over to DP"** is the operational trigger for the entire vacancy-side turn loop (per Jack, decision 19, 2026-07-12; flow-confirmed GlobiFlow UI 2026-07-12). When it flips to "Turn Over to DP" a GlobiFlow chain fires: owner-split record-creation flows (a DP variant gated on Owner = "Dominion Properties" and a parallel 3rd-party variant) that create a Renovation item with **Category = "Turnover"** / "3rd Party - Turnover", the sibling "turnover email from DM to DP" flow, and a Buildertrend job. Its other value, **"Not Turning Over - Parked"**, is the turn-hold decision (see "Parked — not turning over"). CODE GAP: Helm does not read Sched Vacants at all (zero references, per Helm codebase read 2026-07-11), so this trigger — and therefore the whole turn loop — is invisible to Helm today, which is why PH-9/PH-10 don't populate; wiring Sched Vacants in is the net-new work to make turns visible. See "Turnover / turn," "Turnover email," and 09-podio-architecture.md (flow IDs and full trigger/action logic).

**"Handed Off - Rental" (Renovation.Phase value / business PH-3→PH-4 marker)** — Jack's authoritative marker for the renovation→leasing handoff (per Jack, decision 18, 2026-07-12): the point where DP turns a finished property over to DM. Jack's model makes it a **3-part synchronized state change ("the turnover triad")** — (a) the property is a completed job in Buildertrend, (b) the Podio Renovation.Phase is set to "Handed Off - Rental", and (c) the property is added to AppFolio as an asset under the appropriate owner. Any one leg present without the others is a discrepancy (Podio "Handed Off - Rental" but no AppFolio asset = handed to DM but never onboarded; AppFolio asset but Podio still in an in-progress reno phase = mirror lag; Podio "Handed Off - Rental" but BT job not complete = premature flip — see 05-data-sync-map.md and 10-common-data-map.md). Intent vs code: this is Jack's *business* definition of the PH-3→PH-4 transition, but per Helm codebase read 2026-07-11 `derivePhase()` never matches the string, so a Handed-Off property still falls into the PH-3 catch-all in live phase logic — a real gap vs Jack's intent, not just a quirk. The AppFolio-asset-creation is the concrete PH-4 event. For the full phase-pipeline and code-gap treatment see the Renovation-app entry (§2) and 02-property-lifecycle.md; for the authority handoff it triggers see 03-systems-of-record.md.

## 3. Money & accounting terms

**Carry / carry cost / days-not-rented** — Jack's uniform day-cost principle: "The opportunity cost of every day that we own a property that isn't rented is 1 day of rent, or 1 day of carry + opportunity cost of our capital. Every day matters and matters the same." Days-not-rented is the master metric per property across the whole lifecycle; per-phase clocks exist to locate *where* days accumulate, not to rank which delays matter more. See 04-cycle-time-model.md. [confirm: if Helm ever prices a day in dollars, which components count as carry — debt service, taxes, insurance, utilities, plus foregone rent?]

**NOI (Net Operating Income)** — House usage: the monthly amount DM distributes to each owner entity, net of management fees and operating expenses. Note the nuance: property taxes, insurance, and debt service are paid *by the owner entities* out of their own bank accounts after receiving the NOI check — so DM's "NOI" is before those items.

**PMA (Property Management Agreement)** — The formal, market-rate, arm's-length agreement between DM and each property-owning entity. It governs the fee structure and the cash-flow sequence (rent into DM → DM pays opex → monthly NOI distribution to owners).

**Capitalized vs expensed work** — The central accounting split Helm data touches: initial renovation (DP, before the property is placed in service) is fully capitalized into the asset's cost basis; turnover work after a vacancy is typically expensed as repairs & maintenance under DM. DP's monthly corporate overhead allocation to active projects MUST be capitalized. Staff Accountants coordinate with Craig's tracking to route each bill correctly.

**EMD (Earnest Money Deposit)** — The deposit put down on a purchase contract. It is a field on the Buy Closings app (per Podio API tour 2026-07-09) and, per Jack, the accounting of this deposit is the FIRST place a property appears in QuickBooks — during PH-1, before closing. See 02-property-lifecycle.md and 09-podio-architecture.md.

**Settlement statement / ALTA / HUD-1** — The closing document itemizing purchase price and closing costs for an acquisition or disposition. ALTA is the modern form, HUD-1 the legacy form; the team uses the terms loosely for "the settlement statement." Victoria Robinson (Transaction Coordinator) files them in the Buy Closings app (Closings workspace — per Podio API tour 2026-07-09; see §2), and the Controller uses them to establish cost basis in BNA.

**Cost basis** — The capitalized cost of a property for depreciation purposes: purchase price and closing costs (from the settlement statement) plus capitalized renovation costs and capitalized DP overhead. Maintained in BNA by the internal accounting team.

**QRE (Qualified Rehabilitation Expenditure)** — Construction costs on the Howard project that qualify for the historic tax credits. Costs must be coded granularly enough to separate qualifying historic structural work from non-qualifying costs (new additions, certain site work), because that split dictates the credit calculation.

**TIC (tenants in common)** — Woodson and Park Heights are each owned by 3 distinct Delaware LLCs holding title as tenants in common. Operations are fully consolidated: one ledger per property, one tax return at the property-entity level, with the Controller handling annual pro-rata allocations to the TIC LLCs. For Helm's data model, treat each as a single property-owning entity.

## 5. Vendors, labor & AP

**GC / subcontractor / material supplier** — DP acts as General Contractor: it subcontracts labor but purchases materials directly from suppliers to eliminate markup. Subcontractors supply their own materials only when standard and cost-effective (electricians buy wire; HVAC companies buy air handlers). The distinction matters because payment terms differ by category.

**AP calendar (Net 5 labor / 7-day materials)** — Labor & subcontractors are paid strict Net 5: invoices submitted by Monday 9:00 AM are paid that Friday afternoon by ACH, after PMs validate them against open Buildertrend scopes. Material suppliers are paid within 7 days of invoice on rolling weekly runs, regardless of credit lines. Manual checks are being phased out in favor of 100% ACH.

## 6. Helm / app-specific terms

**Phase codes (PH-1 … PH-Z)** — Helm's 12-phase property lifecycle model, derived per property from multi-source signals: PH-1 Acquisition through PH-10 Vacancy/Turn, plus PH-X (Sold/Disposed), PH-Z (Archived/Write-off), and an UNKNOWN bucket when phase can't be determined. Canonical business definitions are in 02-property-lifecycle.md.

**D-rules** — Helm's coded catalog of cross-platform audit rules — 9 in total per Helm codebase read 2026-07-11 (D1, D5, D7, D9, D11, D14, D17, D18, D20; the tour showed only 7 firing, but D5 "write-off still active" and D9 "vacant >14d, no turnover" also exist). The numbering gaps (D2–D4, D6, D8, D10, D12–D13, D15–D16, D19) were never implemented, so those numbers are free for future rules. Discrepancy rules flag disagreements between Podio, AppFolio, and Buildertrend (e.g., D14: "AppFolio application approved but no Podio Vacant Unit entry"); compliance rules (D17/D18) flag license and lead-cert expirations. Business rationale, severity, and resolution playbooks are in 06-audit-rules.md.

**Matched lease** — The unit behind the dashboard's Avg Lease Time KPI (1,902 matched leases at tour time). Strawman reading: an AppFolio lease successfully paired with its corresponding Podio record — the rent-side record is the Lead Certs & Water Bills app (what earlier drafts called "Podio Rental Billing"; per Podio API tour 2026-07-09) — so the lease event can be timed. [confirm: what exactly makes a lease "matched," and what happens to unmatched leases in the KPI?]

**System of record vs derived evidence** — A *system of record* is the platform that is authoritative for a given data element at a given lifecycle stage (see 03-systems-of-record.md for the authority map). Authority is **phase-scoped**: per Jack, decision 18 (2026-07-12), the primary-source custodian moves with the lifecycle — Buildertrend is the source during PH-3 renovation (Podio Renovation mirrors it), and AppFolio is primary post-handoff (PH-4+), with Podio reduced to a compliance supplement. That decides the segment-level custodian; per-field conflict-resolution rules remain deferred (decision 16). *Derived evidence* is Helm's inference: the DERIVED cards in the Lifecycle Phase Journey, each tagged with a date and driver event (e.g., "contract_signed," "move_in") pulled from source signals. When sources conflict, Helm's audit rules flag the discrepancy rather than silently picking a winner.

**Hub snapshot** — Helm's own resolved record of a property's current phase, shown as the "Hub" SNAPSHOT card in the Lifecycle Phase Journey. The Hub is Helm's internal store, distinct from the three source platforms — the resolution of all derived evidence into one answer.

## 7. Systems shorthand

One-liners only — the full what-lives-where map, keying responsibilities, and integration access are in 03-systems-of-record.md.

- **AppFolio** — property management: tenant ledgers, rent rolls, owner accounts, AR. API access in hand.

- **Podio** — operational middleware ("the gap-filler"): 15 workspaces, ~65 apps, mapped first-hand via API on 2026-07-09 — lifecycle-core apps are Buy Closings & Sale Closings (Closings workspace), Renovation (Pipeline - Construction), Vacant Units/Sched Vacants (Occupancy — the live leasing surface; the Occupancy Applications app has been dormant since 2022-05, per Podio API activity probe 2026-07-11), Lead Certs & Water Bills (Rentals Billing & Compliance), plus the older Move Ins & Move Outs generation (an archive, retired ~Nov 2017 per the same probe), 18 per-trade Construction Scheduling apps, and GlobiFlow automations. Full architecture: 09-podio-architecture.md. Jack's stated direction is for Helm to absorb and eventually retire Podio.
- **BNA** — Bloomberg Tax Fixed Assets: depreciation schedules and cost basis, maintained in-house.
- **QuickBooks Desktop → Sage** — the accounting ledgers (one QuickBooks file per tax-filing entity today, migrating to a single Sage database in 2026). Downstream consumers of operational data, not Helm sources; a property's first QuickBooks appearance is the purchase-deposit entry during PH-1, before closing.
