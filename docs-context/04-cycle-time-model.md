# 04 — Cycle-Time Model

This file defines how Helm should think about time: the single master metric, the per-phase clocks that decompose it, and (as a clearly labeled proposal) the violation mechanics that would turn clocks into actionable alerts. Phase definitions live in `02-property-lifecycle.md`; the source-system events referenced below are mapped in `05-data-sync-map.md`; the existing D-rule audit catalog is in `06-audit-rules.md`.

## 1. The philosophy (Jack, verbatim)

> "The opportunity cost of every day that we own a property that isn't rented is 1 day of rent, or 1 day of carry + opportunity cost of our capital. Every day matters and matters the same."

This is the entire model. Everything below is derivation, not addition.

## 2. Design consequences

**(a) The master metric is days-owned-but-not-rented ("carry days") per property.** Jack's phrasing is literal: the cost accrues on every day *we own* a property that isn't rented. Carry therefore accrues from ownership — the `closed` event / PH-2 entry — onward, in every owned non-earning phase. PH-1 (under contract) is pre-ownership: contract-to-close is a cycle-time clock (clock 1 below) but does NOT accrue carry (the [confirm] lives on the carry-day definition below). Terminal phases (PH-X Sold/Disposed, PH-Z Archived/Write-off) are excluded — the clock stops permanently when a property exits the portfolio. Every owned non-earning phase — closing handoff, renovation, listing prep, marketing, application processing, inspection waits, move-in scheduling, turns — is the same currency: one day equals one day of rent.

**(b) There is NO hierarchy of important vs. unimportant phase clocks.** Per-phase clocks exist to locate WHERE days accumulate, not to rank which delays matter. A day lost waiting on a housing-program inspection is worth exactly as much as a day lost to a slow renovation or a listing that didn't go live. Helm should never present one clock as "the critical one" — the correct question is always "which phase is this property's days piling up in right now?"

**(c) Violation detection is therefore two-layered:**
- **Aging-based:** days-in-current-phase vs. the expected duration for that phase (catches the *local* stall).
- **Total-pipeline-based:** cumulative non-earning days for the property's current *pipeline pass* vs. the expected total for that pass (catches death-by-a-thousand-cuts, where every phase is slightly slow but no single clock trips).

Both layers matter because a property can be "on time" in every individual phase and still be 30 days behind in aggregate.

### Definitions

- **Carry day:** one calendar day in which a portfolio property is OWNED (i.e., on or after the `closed` event / PH-2 entry) and in a phase other than PH-8, PH-9 (strawman — see the PH-9 bullet below), PH-X, or PH-Z. PH-1 is pre-ownership: contract-to-close is a cycle-time clock but does NOT accrue carry. PH-1 is not cost-free, though (Jack, 2026-07-09 — `sources/jack-decisions.md` items 7, 9–10): the purchase deposit is tied up from contract (the property's first QuickBooks appearance), and vacant properties — trustee purchases especially — carry BG&E in Dominion's name pre-closing. Real out-of-pocket costs, zero carry days under this definition. [confirm: carry starts at closing, not contract — right?]
- **Pipeline pass:** one contiguous run of non-earning phases. Two kinds:
  - **Acquisition pass:** PH-1 entry → first PH-8 entry (contract → first tenant paying rent). The pass clock spans the whole run, but carry within it accrues only from `closed` / PH-2 entry onward.
  - **Turn pass:** PH-8 exit (move-out) → PH-8 re-entry (next tenant paying rent). May route through PH-10 → PH-5 → PH-6 → PH-7, or back through PH-3 if the turn escalates into a substantial rehab.
- **Earning phase:** PH-8, plus PH-9 as strawman (next bullet).
- **PH-9 (Notice / Pre-vacancy)** — the tenant is typically still paying, so strawman: PH-9 is an EARNING phase; its clock is a scheduling/forecast clock, not carry. The clock still runs because pre-scheduling the turn during notice is how the coming non-earning window gets compressed. **Code reality (per Helm codebase read 2026-07-11, `phase.service.ts`):** PH-9 IS wired (it fires on `unit_status.code IN ('notice_unrented','notice_rented')`) but is masked in practice — `sync_unit`'s status-priority dedupe drops a 'notice' rent-roll row whenever a 'current' row exists for the unit, and `sync_case_podio` then overwrites unit_status from the Vacant Units board, which carries almost no 'notice' states. And **PH-10 is unreachable** — no code path returns it (it exists only in the type union, `PHASE_NAMES`, and a UI swimlane column), so its count is structurally zero forever. So any clock keyed on PH-9 or PH-10 (clocks 8–10 below, and the turn-pass routing above) has essentially no live phase population to run against today; those clocks are computable only from the raw source timestamps described below, NOT from Helm's current derived phase number. ### Clock 1 runs alongside a real work stream — PH-1 is not a passive wait

Per Jack (2026-07-09, `sources/jack-decisions.md` items 6–10): at contract, three records are born near-simultaneously — the Podio **Buy Closings** item, a Podio **Renovation** app item (default status **"Design & Scope"** when vacant; **"Trash Out"** may be added when there's abandoned personal property), and the QuickBooks entry for the purchase deposit. Vacant properties get BG&E in Dominion's name so they can be scoped, and on trustee (courthouse-steps) purchases found vacant, locks are changed and trash-out happens BEFORE closing. The renovation *planning* clock therefore effectively starts at contract, inside clock 1 — clock 1 measures closing logistics while scope/design (and, for trustee-vacant deals, trash-out) runs in parallel. None of this changes the carry-day strawman (carry still starts at closing); it means PH-1 already incurs real out-of-pocket costs (deposit tied up, BG&E) without accruing a single carry day.

### Clock 2's expectation forks by seller type

Jack's seller-type fork (item 10) means closing→job-start cannot have one honest target:

- **Trustee (courthouse-steps) purchases found vacant** are treated as abandoned — scoped, locks changed, and trashed out BEFORE closing — so clock 2 should run near-zero: Construction can start almost immediately after settlement.
- **Private-individual-seller purchases** get no Dominion utilities and no trash-out until AFTER closing, so their clock 2 legitimately contains post-close scoping days.

A single ≤ 7-day strawman under-challenges trustee-vacant deals and may unfairly penalize private-seller ones. [confirm: should clock-2 targets be split by acquisition type (trustee-vacant vs private-seller)?] The attribute already exists in the source data: Podio Buy Closings carries a **Type** field (Private | Trustee | REO | Builders | Short Sale | OHAAT | NCST), per Podio API tour 2026-07-09 — so splitting targets requires no new data capture. Whether Helm should surface acquisition type as a first-class property attribute is [confirm]'d in `02-property-lifecycle.md`.

### The one live datapoint

The Helm dashboard already shows **"Avg Lease Time 62.6d" (↓2d), computed from 1,902 matched leases**. This is the only cycle-time number in production today. **Its definition is now RESOLVED from code (this supersedes the earlier [confirm] — per Helm codebase read 2026-07-11):** `days = AppFolio occupancy.move_in_date − COALESCE(Podio vacant_unit.marketing_start_date, intake_date)`, using the most-recent qualifying Podio date ≤ the move-in. So the span is **marketing-start → move-in** (falling back to Podio intake_date when no marketing date exists) — closest to catalog clocks 5–7 combined; it is NEITHER listing→move-in NOR vacancy→lease-signed. Population = **every** `hub.occupancy` row that has a move_in date (any status), **ALL-TIME with no rolling window**, filtered to a 0–365-day outlier band; "1,902 matched leases" is the surviving row count from a live inner LATERAL join to a qualifying Vacant Unit record (unmatched occupancies silently drop). Two data-quality caveats baked into that number: rows are per (occupancy, tenant), so **co-tenants double-count**, and the `property_id` fallback lets one unit borrow another unit's marketing date. The **"↓ 2d" trend is a hardcoded string, not a real month-over-month computation** — as are all the dashboard trend chips (no MoM math exists). Do not treat 62.6d as a benchmark for any single catalog clock: it is a marketing→move-in composite, not a single clock edge.

## 4. PROPOSAL — violation mechanics

*Everything in this section is a proposal for Jack's reaction, not a spec.*

**Code status (per Helm codebase read 2026-07-11) — NONE of this section exists in code yet.** There are NO T-rules (the rule id is a free-form string, but every implemented rule is a D-rule — no `T*` prefix exists anywhere), NO aging buckets, NO `days_in_current_phase` / `carry_days` fields surfaced anywhere, and NO month-over-month trend computation (the dashboard chips are hardcoded). The carry-day / T-rule work is **unblocked but unbuilt**: the substrate `hub.property_phase_history` — a nightly snapshot recording per-phase open/close intervals and `days_in_phase`, written by the 03:00 snapshot job — already exists, but **nothing aggregates it** (the dashboard doesn't even read it). The scaffolded stage-aging columns `hub.cases.is_stale` / `hub.cases.days_in_stage` are **dead write-once columns** (always `false` / `0`) — stage-aging was scaffolded and never built. Separately, **findings-aging is impossible today**: audit findings are recomputed live on every read and never persisted (no dismiss / snooze / resolve, `detected_at` = request time), so there is no stored finding history to age against — any T-rule/aging layer over *findings* would first require a persistence layer that does not exist. Aging of *phases* is the tractable near-term path, because `property_phase_history` already captures the intervals; aging of findings is not. Note also that Helm is **read-only inbound** with no notification/alerting engine today, so any T-rule that emits an alert (or replaces a GlobiFlow watchdog, item 3 below) is net-new outbound machinery, not a re-wire of something Helm already does.

1. **Per-property days-in-phase counter.** Every property in a non-terminal phase carries a visible `days_in_current_phase` (and `carry_days_this_pass` for the current pipeline pass). These are first-class fields on the registry row and detail page, not report-only aggregates.

2. **Aging buckets, uniform across clocks.** Because no clock outranks another, use the same relative bands everywhere: **ON-TRACK** (< 1.0× expected), **AGING** (1.0–1.5×), **STALLED** (1.5–2.0×), **CRITICAL** (> 2.0×). [confirm: are these multipliers right, or should bands be absolute day counts per clock?]

3. **A T-rule namespace parallel to the D-rules.** D-rules catch cross-system *data* discrepancies; T-rules catch *time* violations. Strawman catalog:
   - T1–T10: per-clock aging violations, numbered to match the clock catalog (T1 = acquisition→closing overdue, T9 = turn overdue, etc.).
   - T20: pipeline-pass total exceeded (aging-based layer didn't trip but cumulative carry days for the pass is over target).
   - T30: phase UNKNOWN for > N days — a property whose phase can't be resolved has an unmeasurable clock, which is itself a violation. [confirm: N?]
   - T31: "parked" property review — see hold-state question below.
   Severity derives from the aging bucket (AGING → LOW, STALLED → MEDIUM, CRITICAL → HIGH), so severity encodes *how late*, never *which clock* — preserving consequence (b).

   **Prior art — Podio already runs deadline watchdogs.** Per GlobiFlow UI tour 2026-07-09 (`sources/podio-architecture.md` §6), three GlobiFlow flows are already doing primitive T-rule work today: **"notify Leasing and property manager that RFTA submission is late"** (Vacant Units — a deadline alert inside clock 7's housing-program wait), **"Upcoming LLF Expiration Date - WIP DM"** (Lead Certs & Water Bills — a lead-cert expiry watchdog, proto-D17/D18 territory; see `06-audit-rules.md`), and **"Email Craig when inspection is less than 15 days out and work is not reported completed by contractor"** (Reinspections). This is direct evidence the business already wants deadline alerting — T-rules aren't a Helm invention, they're uniform machinery for a class of alerts the team has been hand-building one flow at a time — and each of these hand-built watchdogs is a candidate for one-for-one replacement by a T-rule. [confirm: when T-rules land, should Helm replace these three GlobiFlow watchdogs outright (retire the flows), or run in parallel until the T-rule engine is trusted?]

4. **Dashboard "carry days at risk" rollup.** One number for the whole portfolio: Σ over all properties currently in owned non-earning phases (per the carry-day definition in §2) of their carry days this pass — i.e., properties × days in owned non-earning phases. Optionally priced in dollars as carry days × per-property daily market rent. [confirm: for the dollarized version, use AppFolio listed market rent, last in-place lease rent, or a flat portfolio average?] This is the headline the philosophy demands: the total rent currently being forfeited to pipeline time, updated daily.

5. **Hold states.** Statuses like "parked - not turning over" (observed in the registry) are deliberate holds, and the decision point is now located: parking is keyed at **Sched Vacants.Turn Over Status = "Not Turning Over - Parked"** (the PH-9 notice record), with the same hold appearing as "parked - not turning over" in Vacant Units.Status and as "Parked" values in Renovation and Lead Certs — a cross-app hold vocabulary, per Podio API tour 2026-07-09 (`09-podio-architecture.md`). **Code reality (per Helm codebase read 2026-07-11):** Helm does NOT read Sched Vacants (zero references to it or its app id anywhere in the repo), so the `Sched Vacants.Turn Over Status = "Not Turning Over - Parked"` key is unavailable to Helm today. The parked signal Helm *can* see comes from the two apps it does ingest: `Vacant Units.Status = "parked - not turning over"` and `Renovation.Phase = "Parked"` — and the Renovation "Parked" value is already caught by audit rule **D5 ("write-off/parked while still active", MED)**, one of the two rules (D5, D9) that exist in code but were not seen firing on the tour (see `06-audit-rules.md`). So a parked-property treatment can be built on ingested data today — just off Vacant Units + Renovation, not off the Sched Vacants field. Strawman: a parked property is *excluded from T-rule escalation* (nobody is failing) but *still accrues carry days in the rollup* (the economic cost is real and shouldn't be hidden).
